Overview\n\n### 1.1 Purpose\n\nThe **LRU Prompt Cache** is a performance optimization system for the MLX OpenAI Server that reduces computational overhead in transformer-based language model inference by reusing previously computed **KV (key-value) caches** from the model's attention mechanism.\n\nWhen generating text, transformer models compute attention over all previous tokens, storing intermediate **key** and **value** vectors in a KV cache. For prompts that share common prefixes (e.g., repeated system prompts, similar user queries), the server can reuse these cached attention states instead of recomputing them, resulting in significant speed improvements—especially for long prompts.\n\n### 1.2 Scope\n\nThe cache is **only enabled for language models** (`model_type = \"lm\"`). Each model handler instance maintains its own independent cache; in multi-model mode, each model gets a separate cache.\n\n---\n\n## 2. Core Data Structures\n\nThe cache is implemented in `app/utils/prompt_cache.py` as the `LRUPromptCache` class. It combines three key structures:\n\n### 2.1 Trie (Prefix Tree)\n\nThe trie stores token sequences as nested dictionaries:\n\n```\n_cache: dict[int, Any]\n```\n\n- Each node represents a token ID\n- Child nodes are stored as `node[token_id] = child_dict`\n- A node may contain a `\"cache\"` key holding a `CacheEntry`\n- This structure enables **efficient prefix matching**: given a sequence of token IDs, we can traverse the trie to find the longest prefix that has a cached KV state\n\n**Example:**\n```\n_cache = {\n    123: {  # token \"Hello\"\n        456: {  # token \",\"\n            789: {  # token \"how\"\n                \"cache\": CacheEntry(...)  # cached KV for \"Hello, how\"\n            },\n            101: {  # token \"hi\"\n                \"cache\": CacheEntry(...)  # cached KV for \"Hello, hi\"\n            }\n        }\n    }\n}\n```\n\n### 2.2 LRU Queue\n\n```\n_lru: deque[tuple[int, ...]]\n```\n\nA double-ended queue tracking cache keys in **least-recently-used order**:\n- New entries are appended to the right (most recently used)\n- When capacity is exceeded, entries are popped from the left (least recently used)\n- Each element is a **tuple** of token IDs (hashable) corresponding to a cache key\n\n### 2.3 CacheEntry\n\n```python\n@dataclass\nclass CacheEntry:\n    prompt_cache: list[Any]  # The actual KV cache data from the model\n    count: int               # Reference count (supports sharing)\n```\n\n- `prompt_cache`: The opaque KV cache structure returned by `model.create_prompt_cache()` and consumed by the model during generation\n- `count`: Tracks how many active requests are using this cache. Design Rationale\n\n### 5.1 Why a Trie?\n\n- **Prefix matching**: The fundamental operation is \"find the longest cached prefix of this prompt\". A trie provides O(L) lookup where L is prompt length, independent of cache size.\n- **Shared prefixes**: Multiple prompts that share beginnings (e.g., \"Write a poem about X\", \"Write a story about Y\") naturally share trie nodes, saving memory.\n- **Flexible reuse**: Supports three match types (exact, shorter, longer) with a single structure.\n\n### 5.2 Why LRU Eviction?\n\n- **Bounded memory**: On Apple Silicon, memory is limited. LRU ensures the cache doesn't grow indefinitely.\n- **Temporal locality**: In practice, recent prompts are more likely to be reused (e.g., iterative refinement, conversational turn-taking). LRU captures this pattern.\n- **Simplicity**: LRU with a deque is straightforward to implement and has O(1) operations per insert/evict.\n\n### 5.3 Reference Counting\n\nAllows **safe sharing** of cache entries across concurrent requests:\n- Multiple requests can fetch the same cache (e.g., identical prompts)\n- The first request to complete will **extract** (remove) the cache if its count is 1\n- Subsequent requests still hold references (count > 1) and will get a **deep copy** when they extract\n- This prevents a race where two requests try to update the same cache entry simultaneously\n\n**Example:**\n```\nCache entry K with count = 2 (two requests using it)\nRequest A finishes → extract() sees count=2 → decrement to 1, return copy\nRequest B finishes → extract() sees count=1 → delete original, return original\ninsert_cache(K_modified) → creates new entry\n```\n\n### 5.4 Trimming Support\n\nWhen a cached sequence **extends** the current prompt (longer match), we can \"trim\" the cache to the shared prefix instead of discarding it entirely. This requires cooperation from the underlying MLX model's cache structure (`mlx_lm.models.cache`).\n\n**Benefit:** If we have cached KV for tokens `[A, B, C, D]` and the new prompt is `[A, B, X]`, we can:\n- Reuse the KV for `[A, B]` from the longer cache\n- Trim away the extra `[C, D]` part\n- Avoid creating a completely fresh cache\n\nThis captures cases where a previous generation was longer than the current prompt, but they share a prefix.\n\n---\n\n## 6. Usage in MLXLMHandler\n\n### 6.1 Initialization\n\n```python\nself.prompt_cache = LRUPromptCache(max_size=prompt_cache_size)\n```\n\nThe cache size is set via `--prompt-cache-size` (default 10). Each handler instance has its own cache.\n\n### 6.2 Request Handling\n\nBoth `generate_text_stream` and `generate_text_response` follow the same pattern:\n\n```python\n# 1. Generate (streaming or blocking)\nresponse = self.inference_worker.submit(...,\n    input_ids=rest_input_ids,\n    prompt_cache=cache,\n    ...\n)\n\n# 4. Update cache key with generated tokens\ncache_key += response.tokens  # or accumulate from chunks\n\n# 5. Insert updated cache\nself.prompt_cache.insert_cache(cache_key, cache)\n```\n\n**Key points:**\n- The model is **never called with the full `input_ids`**; only the uncached remainder (`rest_input_ids`)\n- The cache object is mutated during generation (KV cache grows as new tokens are produced)\n- After generation, the cache represents the KV state for the **full** sequence `cache_key`\n- Inserting with the full key updates the trie path appropriately (replacing evicted entries along that path if they existed)\n\n### 6.3 Debug Logging\n\nWhen `--debug` is enabled, the handler logs:\n- Cache hit statistics (cached tokens vs. remaining tokens)\n- Raw responses and generation metrics\n- Parser events (for reasoning/tool parsing, not directly cache-related)\n\n---\n\n## 7. Performance Characteristics\n\n### 7.1 Time Complexity\n\n- **`_search`**: O(L) where L = length of `tokens_ids`. Single pass down the trie, plus optional DFS for longer match (worst-case O(total_cached_nodes) but cache size is bounded).\n- **`fetch_nearest_cache`**: O(L) dominated by `_search`\n- **`insert_cache`**: O(L) to traverse/create path + O(1) LRU operations + potential O(L) eviction (delete path)\n- **`_delete`**: O(L) to walk up and prune\n\nAll operations are effectively constant-time for typical prompt lengths (hundreds to thousands of tokens) because the cache size (max 10 by default) bounds the trie depth and breadth.\n\n### 7.2 Space Complexity\n\n- **Trie**: Each node is a dict mapping token IDs to child dicts. In worst case (no shared prefixes), each cache entry adds L nodes where L is sequence length. With sharing, space is reduced.\n- **LRU deque**: O(N) where N = number of cache entries (≤ max_size)\n- **CacheEntry**: The `prompt_cache` is the largest component, holding model-specific KV tensors. Its size is roughly O(context_length × num_layers × hidden_dim × 2). With `prompt_cache_size = 10`, at most 10 such structures coexist.\n\n**Memory bound:** The `max_size` parameter directly limits the number of KV cache objects. However, the trie itself (token IDs) is small compared to the KV tensors.\n\n### 7.3 Trade-offs\n\n| Aspect | Design Choice | Reasoning |\n|--------|--------------|-----------|\n| Match policy | Prefer exact > shorter > longer | Maximizes reuse; longer requires trim support |\n| Shorter threshold | `last_cache_index > 0` (depth ≥ 2) | Avoids degenerate 1-token hits |\n| Eviction | Strict LRU | Simple, works well for temporal locality |\n| Reference counting | Manual (increment on insert, decrement on extract) | Enables safe sharing across concurrent requests |\n| Cache key | Full sequence (prompt + generation) | Ensures complete coverage for future exact matches |\n| Insert behavior | Overwrite existing path | If same prefix re-appears, new generation updates it |\n\n---\n\n## 8. This was fixed by using the **full `input_ids`** as the cache key.\n\n**Correct:**\n```python\ncache_key = input_ids[:]  # Full prompt\n# after generation:\ncache_key += generated_tokens\ninsert_cache(cache_key, cache)\n```\n\n**Incorrect (leaky):**\n```python\ncache_key = rest_input_ids  # Only uncached suffix\n```\n\n### 8.2 Trimming Requirement\n\nThe \"longer match\" path only succeeds if `can_trim_prompt_cache(cache_entry)` returns True. Not all model cache implementations support trimming. If trimming is unsupported, the longer match is ignored and the system falls back to no cache.\n\n### 8.3 Concurrency Model\n\n- Each `MLXLMHandler` instance has **one** `LRUPromptCache`\n- Requests are processed **sequentially** per model instance because they go through an `InferenceWorker` with a bounded queue and limited concurrency (`max_concurrency`)\n- The cache itself has **no internal locking**; it's safe because only one request accesses it at a time (the inference worker processes requests one-by-one for a given handler)\n- If multiple handler instances are created (e.g., multi-model mode with the same model ID), each has its own cache—no sharing across processes\n\n### 8.4 Cache Invalidation\n\nThere is **no explicit invalidation**. Entries are evicted automatically by LRU when capacity is exceeded. There is no time-based expiration or manual purge.\n\n### 8.5 Empty Cache After Extract\n\nWhen `extract()` removes the last reference to a cache entry, it also deletes the corresponding leaf node from the trie and prunes empty ancestors.
